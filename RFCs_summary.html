Introduction

They are the RFCs that define the IRC protocol.
In this summary, we will only cover the most important parts of the RFCs
that are relevant to the implementation of our own IRC server.
That means that everything related to the client side of the protocol
and to the server-to-server communication will be voluntarily ignored.

Note that we are using the Augmented Backusâ€“Naur form to represent
the grammar rules that apply to the different components of the IRC protocol.
However, we also provided a more human-readable description for each rule.

Server

Name

Version

Users

Nickname

Nicknames are the names used by the users on the IRC network
to identify themselves and each other, for example when you want
to send a private message to someone, or to invite someone to a channel, etc...

Hostname

Hostnames are the names of the hosts that the users are running on.
For example, if a user is running on a computer named "mycomputer",
then their hostname will be "mycomputer".

Username

Usernames are the names of the users on their host.
For example, if a user is logged as "John" on their computer,
then their username will be "John".

Realname

Realnames are the real names of the users.
For example, if a user's real name is "John Doe",
then their realname will be "John Doe".

Modes

User modes are flags that indicate different states about the user.
In messages sent and received by the server, user modes are represented by a single letter.

Mask

A mask is a way to identify a user without any possible ambiguity.

Channels

Name

Topic

Topics are a short description about the channel. Most of the time,
it describes the main subject that is discussed in the channel.

Note that the topic may be an empty string,
in which case it is assumed that the channel has no topic.

Modes

Channel modes are flags that indicate different states about the channel.
Unlike the user modes, some channel modes may be related to some users,
and not only to the channel itself. Therefore, channels may need to have extra fields
to keep track of the users that are related to these modes (e.g. a list of channel operators)
In messages sent and received by the server, channel modes are represented by a single letter.

Parameters

Some channel modes require some parameters. Among the channel modes listed above,
there are those that require some parameters:

  • +b : takes 1 or more user masks to add to the banlist of a channel
  • -b : takes 1 or more user masks to remove from the banlist of a channel
  • +k : takes 1 key to set for a channel, rejecting new users to join it
    unless they provide the correct key.
  • +l : takes 1 number to set the maximum number of users present in a channel
  • +o : takes 1 or more user masks to add to the chanoplist of a channel
  • -o : takes 1 or more user masks to remove from the chanoplist of a channel

Messages

Prefix

The purpose of the prefix is to explicitly indicate the true origin of a received message.
However, the prefix is optionnal, and thus, if a received message has no prefix,
it is assumed that its true origin is the connection from which it was received.

A prefix may be invalid for 3 different reasons, and in each case,
the received message that contain this invalid prefix must be discarded,
and an error must be sent back to the client from which the message was received.

Parameter list

Commands

Commands are the main way to communicate with the server.
They are used to perform several actions such as sending messages,
joining channels, changing nicknames, etc...

There is a list of the different commands that we must implement below.

PASS

The PASS command can be used only during the registration process.
It is an optional command, depending on whether the server
that the client tries to connect to is password-protected or not.
The PASS command can only be sent once per connection.
When the server receives a PASS command, it must check
if the given password matches the configured one.
If it does, then the connection is authenticated.
Otherwise, the connection is closed.

Parameters

password
1 password

Replies

  • ERR_NEEDMOREPARAMS
  • ERR_ALREADYREGISTERED

NICK

The NICK command is used to change the nickname of a user.
When the server receives a NICK command, it must check
if the given nickname is valid and not already in use.
If it is, then the nickname of the user is changed to the given one.
Otherwise, an appropriate error reply is sent back to the client.

Parameters

nick
1 nick

Replies

  • ERR_NONICKNAMEGIVEN
  • ERR_ERRONEUSNICKNAME
  • ERR_NICKNAMEINUSE

USER

The USER command must be used only during the registration process.
It must be sent only once per connection.
When the server receives a USER command, it must check
if the given username and hostname are valid.
If they are, then the username and hostname of the user are set to the given ones.
Otherwise, an appropriate error reply is sent back to the client.

Parameters

user space umode-mask space host space %x3a realname
  1. Start with 1 user
  2. Contain 1 space
  3. Contain 1 umode-mask
  4. Contain 1 space
  5. Contain 1 host
  6. Contain 1 space
  7. Contain 1 octet that has the value 0x3a (:)
  8. Contain 1 realname

The umode-mask component is a numeric value
that represents the initial user modes of the user.

This numeric value is actually a bit mask,
where each bit represents a specific user mode.
However, only 2 of them are able to be set by the USER command:

  • the bit at the index 2, which represents the `i` user mode
  • the bit at the index 3, which represents the `w` user mode

For example, if the umode-mask component is "8",
8 being 00001000 in binary, then the `w` user mode will be set,
while the `i` user mode will not.

Replies

  • ERR_NEEDMOREPARAMS
  • ERR_ALREADYREGISTERED

OPER

The OPER command is used to authenticate a user as an IRC operator.
When the server receives an OPER command, it must check
if the given username-password association matches any
of the configured ones.
If it does, then the server must send back a RPL_YOUREOPER
to the client, followed by a MODE command to indicate them their new modes.

Parameters

user space password
  1. Start with 1 user
  2. Contain 1 space
  3. Contain 1 password

Replies

  • RPL_YOUREOPER
  • ERR_NEEDMOREPARAMS
  • ERR_PASSWDMISMATCH
  • ERR_NOOPERHOST

MODE

The MODE command is a dual purpose command in IRC.
It is used to modify both user modes and channel modes.
When the server receives a MODE command, it must first determine
whether it is targeting a user or a channel.

User

If the target is a user, then the server must check
that both the given nickname and the sender nickname are the same.
If they are, then the server must finally check
that the given mode changes are valid.
If they are, then the server must apply them.
Otherwise, an appropriate error reply is sent back to the client.

Parameters

nick space *( ( %x2b / %x2d ) *umode )
  1. Start with 1 nick
  2. Contain 1 space
  3. Contain 0 or more pattern, which must:
    1. Start with 1 octet that has either the value 0x2b (+)
      or the value 0x2d (-)
    2. Contain 0 or more umode

Replies

  • RPL_UMODEIS
  • ERR_NEEDMOREPARAMS
  • ERR_UMODEUNKNOWNFLAG
  • ERR_USERSDONTMATCH

Channel

If the target is a channel, then the server must check
that the given channel exists.
If it does, then the server must check
that the sender is either one of its channel operator or its creator.
If they are, then the server must finally check
that the given mode changes are valid.
If they are, then the server must apply them.
Otherwise, an appropriate error reply is sent back to the client.

Parameters

channel space *( ( %x2b / %x2d ) *cmode *( space cmode-param ) )
  1. Start with 1 channel
  2. Contain 1 space
  3. Contain 0 or more pattern, which must:
    1. Start with 1 octet that has either the value 0x2b (+)
      or the value 0x2d (-)
    2. Contain 0 or more cmode
    3. Contain 0 or more pattern, which must:
      1. Start with 1 space
      2. Contain 1 cmode-param

Replies

  • RPL_CHANNELMODEIS
  • RPL_INVITELIST
  • RPL_ENDOFINVITELIST
  • RPL_BANLIST
  • RPL_ENDOFBANLIST
  • ERR_USERNOTINCHANNEL
  • ERR_NEEDMOREPARAMS
  • ERR_KEYSET
  • ERR_UNKNOWNMODE
  • ERR_NOCHANMODES
  • ERR_CHANOPRIVSNEEDED

QUIT

The QUIT command is used to disconnect a client from the server.
When the server receives a QUIT command,
it must send back an ERROR message to the client,
to confirm that they are about to be disconnected.
Then, it must close the connection with the client,
remove them from every channel they were in, and then
remove them from its list of known clients.
Finally, the server must send a QUIT message
to every channel that the user was in,
to inform them of the user's departure.

Note: for each ERROR and QUIT message sent by the server,
the server must pass in parameter either the given quit message if there is one,
or a default one if there is not.

Parameters

[ quit-message ]
0 or 1 pattern, which must:
  1. Start with 1 octet that has the value 0x3a (:)
  2. Contain 1 quit-message

Replies

None

JOIN

The JOIN command is used to make a user join a channel.
When the server receives a JOIN command, for every given channel,
it must do the following routine:

Check that the given channel exists.
If it does, then check that the user isn't already in it.
If they are, then stop the routine here.
Else, check that the given channel is neither full,
nor marked as invite-only.
If it isn't, then check that the user hasn't reached
the maximum number of channels they can join yet.
If they haven't, then check that the user isn't banned
from the channel.
If they aren't, and if the given channel has the `k` mode set,
then check that the user provided the correct key.
If they did, then add the user to the channel,
and finally, send back the following to the client in this order:

  1. A JOIN message, to confirm that the user have successfully joined the given channel
  2. A RPL_TOPIC to indicate the topic of the joined channel
  3. The result of the NAMES command with the joined channel
    passed in parameter
Otherwise, an appropriate error reply is sent back to the client.

Parameters

channel *( %x2c channel ) [ space key *( %x2c key ) ] / %x30
Either 1 of the following:
  • 1 pattern, which must:
    1. Start with 1 channel
    2. Contain 0 or more patterns, which must:
      1. Start with 1 octet that has the value 0x2c (,)
      2. Contain 1 channel
    3. Contain 0 or 1 pattern, which must:
      1. Start with 1 space
      2. Contain 1 key
      3. Contain 0 or more patterns, which must:
        1. Start with 1 octet that has the value 0x2c (,)
        2. Contain 1 key
  • 1 octet that has the value 0x30 (0)

Replies

  • RPL_TOPIC
  • ERR_NOSUCHCHANNEL
  • ERR_TOOMANYCHANNELS
  • ERR_NEEDMOREPARAMS
  • ERR_CHANNELISFULL
  • ERR_INVITEONLYCHAN
  • ERR_BANNEDFROMCHAN
  • ERR_BADCHANNELKEY
  • ERR_BADCHANMASK

PART

The PART command is used to make a user leave a channel.
When the server receives a PART command, for every given channel,
it must do the following routine:

Check that the given channel exists.
If it does, then check that the user is in it.
If they are, then remove the user from the channel,
send back a PART message to the client,
to confirm that they have successfully left the given channel,
and finally, send a PART message to every user
that are present in the given channel,
to inform them of the user's departure.
Otherwise, an appropriate error reply
is sent back to the client.

Note: for each PART message sent by the server,
the server must pass in parameter either the given part message if there is one,
or a default one if there is not.

Parameters

channel *( %x2c channel ) [ space part-message ]
  1. Start with 1 channel
  2. Contain 0 or more patterns, which must:
    1. Start with 1 octet that has the value 0x2c (,)
    2. Contain 1 channel
  3. Contain 0 or 1 pattern, which must:
    1. Start with 1 space
    2. Contain 1 part-message

Replies

  • ERR_NOSUCHCHANNEL
  • ERR_NOTONCHANNEL
  • ERR_NEEDMOREPARAMS

TOPIC

The TOPIC command is used to set, unset, or change the topic of a channel.
When the server receives a TOPIC command, it must check that the given channel exists.
If it does, then the server must check that the user is in it.
If they are, then the server must check if there is a given topic to set.

  • If there is, then the server must check if the `t` channel mode is set.
    If it is, then the server must check that the user is a channel operator.
    If they are, or if the `t` channel mode is not set,
    then the server must set the topic of the channel to the given one,
    and finally, send back a TOPIC message to every user
    that are present in the given channel,
    to inform them of the new topic of the channel.
  • If there is not, then the server must check whether the current topic
    of the given channel is an empty string.
    If it is, then the server must send back a RPL_NOTOPIC
    to the client, to indicate that the given channel has no topic.
    Else, the server must send back a RPL_TOPIC
    to the client, to indicate the topic of the given channel.
Otherwise, an appropriate error reply is sent back to the client.

Note: for each TOPIC message sent by the server,
the server must pass in parameter the given topic.

Parameters

channel [ " :" topic ]
  1. Start with 1 channel
  2. Contain 0 or 1 pattern, which must:
    1. Start with the string " :"
    2. Contain 1 topic

Replies

  • RPL_NOTOPIC
  • RPL_TOPIC
  • ERR_NOTONCHANNEL
  • ERR_NEEDMOREPARAMS
  • ERR_CHANOPRIVSNEEDED

Replies

Most of the messages that the server receives from a client
generate a reply to that client from the server to indicate
whether the operation succeeded or failed.

Replies are not allowed to originate from a client.
Any such messages received by the server must be silently discarded.

There is a list of the different replies that we must implement need below.
